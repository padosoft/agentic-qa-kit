# ADR-004 — Zod as source of truth, JSON Schema as a generated artifact

- **Status:** Accepted
- **Date:** 2026-05-17
- **Tags:** schema / dx

## Context

Every persisted data shape needs to live in (a) runtime validation, (b)
TypeScript types, (c) a portable schema other tools and language ports can
consume. Authoring three separate copies guarantees drift.

## Decision

The Zod schemas under `packages/schemas/src/*.ts` are the source of truth.

- TypeScript types come from `z.infer<typeof X>`.
- JSON Schema (Draft 2020-12) is generated by a build step
  (`scripts/emit-json-schemas.mjs`) from Zod via `zod-to-json-schema`, with
  post-emit patches for Draft 2020-12 compliance (numeric `exclusiveMinimum`)
  and for `superRefine` mirroring where feasible (Finding's verified gating,
  Run's terminal-state gating).

`schemas/v1/*.schema.json` is shipped (in the package's `files` allowlist) so
LSP / non-TS consumers get the schema; it is not authored by hand and is
excluded from biome's formatter so the build is idempotent.

## Consequences

- One definition, two derived artifacts. Drift is impossible.
- Cross-field rules that JSON Schema can't express (e.g. `finished_at >=
  started_at`) live only on the Zod side; the gap is documented in a
  `$comment` and tracked in #3.

## Alternatives considered

- **TypeBox.** Rejected: smaller ecosystem and weaker error messages.
- **Hand-authored JSON Schema with TS types generated from it.** Rejected:
  JSON Schema is harder to read/refactor than Zod for the team that owns
  these definitions.
